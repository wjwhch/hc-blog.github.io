    # JSX的本质是什么？

    Javascript语法的扩展。充分具备JS的能力。

    # JSX是语法是如何在Javascript中生效的：Babel

    JSX会被Babel编译为React.createElement()，该函数将返回一个叫做"React Element"的JS对象。

    # 什么是Babel？

    是一个工具链，能够将ECMA2015+语法编写的代码转换为向后兼容的JS语法，以便能够运行在但钱和旧版本的浏览器或其他环境中。

    # React.createElement(type, config, children)

    接受三个参数，这三个参数囊括了创建一个元素所需要全部信息。

    • type: 用于标识节点的类型。它可以是类似“h1”“div”这样的标准 HTML 标签字符串，也可以是 React 组件类型或 React fragment 类型。

    • config：以对象形式传入，组件所有的属性都会以键值对的形式存储在 config 对象中。

    • children：以对象形式传入，它记录的是组件标签之间嵌套的内容，也就是所谓的“子节点”“子元素”。

      createElement 函数就是做了一层数据操作，格式化数据。最终调用 ReactElement() 实现元素的创建。

      ReactElement 返回的对象实例，本质上是以Javascript对象形式存在的对DOM的描述。（虚拟DOM）

    虚拟DOM到真实DOM依靠ReactDom.render()。

    虚拟DOM -----> ReactDom.render() -----> 真实DOM

    # ReactDom.render(ele, contaniner, [callback])

    接受三个参数，其中第二个参数就是一个真实的DOM节点，充当容器的角色，React元素最终会被渲染到容器内。

    # react两次求变的原因：Fiber架构

    Fiber会使原本同步的渲染变异步。

    16之前，每次的组件更新都会生成新的虚拟DOM树，通过与前一个虚拟DOM树进行diff，实现DOM定向更新。是一个递归的过程，当递归调用栈很深时无法及时返回，整个过程漫长不可断。同步渲染一旦开始直到递归彻底完成前都占据主线程，无法处理渲染以外的事，会造成页面卡顿，用户交互无反应等问题。

    16之后的Fiber架构，会将一个大的更新任务拆分成若干小任务，每完成一次小任务，打断渲染进程，处理优先级更高的任务。这就是异步渲染。

    打断是有原则的：生命周期被分为了render阶段和commit阶段。commit阶段又分为pre-commit和commit。

    • render阶段： 纯净且没有副作用，可能会被React暂停、终止或者重新启动。
    • pre-commit阶段：可以读取DOM。
    • commit阶段：可以使用DOM，运行副作用，安排更新。

    为什么render阶段能打断，因为此过程用户不可见，而commit阶段涉及DOM的操作，不能明目张胆的更改。

    # 废旧立新的思考：

    Fiber机制下render阶段能被暂停、终止和重启。重启时是***重新执行一整遍任务***而不是从暂停处继续，这就导致render阶段的生命周期有可能重复执行。所以在render阶段的生命周期内不要做骚操作。

    生命周期的变化是为了配合Fiber的异步渲染机制，改造过程中精益求精，针对生命周期被滥用的部分推行了具有强制性的最佳实践。首先确保了Fiber机制下数据和视图的安全性，同时也确保了生命周期方法的行为更加纯粹、可控、可预测。

    # React的组件间的数据流动

    react视图会随着数据的变化而变化

    # 基于props的单向数据流

    所谓单向数据流，就是指当前组件的state以props的形式流动时，只能流向组件树中比自己层级更低的组件。

    • 父-子：props
    • 子-父：由于单向数据流，子不能直接传父，所以父可以传一个函数给子，子通过调用函数传入state
    • 兄弟组件：数据都通过props由父组件传入，同时传入修改state的方法，兄弟组件调用方法传入state

    当父子兄弟之间层级很深时，使用发布订阅模式、context或者redux。

    Context API ：新API解决了当父组件shouldComponentUpdate组织更新时，子组件不会更新。

    Redux ：

    是Javascript状态容器，它提供可预测的状态管理

    redux由三部分组成：store、reducer、action

    • store：单一的数据源，而且是只读的。
    • action：是对变化的描述。
    • reducer：是一个函数，负责对变化进行分发和处理，最终将新的数据返回store。

    三者之间紧密配合，形成redux独树一帜的工作流。

    Redux在整个过程中，数据流是严格单向的。

    # Hooks

    Hooks的出现是react团队对类组件和函数组件两种形式的思考和侧重。是一套能够使函数组件更强大、更灵活的钩子。

    # 类组件

    基于ES6 Class 这种写法，通过继承React. Component得来的React组件。

    # 函数组件/无状态组件

    就是以函数形式存在的React组件，早期没有Hooks的加持，函数组件内部无法定义和维护state，因此还有一个别名叫无状态组件。

    # 函数组件和类组件的不同

    • 类组件需要继承class，函数组件不需要
    • 类组件可以访问生命周期方法，函数组件不行
    • 类组件可以获取实例化后的this，并基于这个this做各种各样的事，而函数组件不行
    • 类组件可以定义并维护state，函数组件不行。
    • ......

    函数组件可以捕获render内部的状态，这是两类组件最大的不同

    在线Demo (opens new window) ，这个demo说明，函数组件能够捕获render内部的状态。而class不行

    （为什么demo内的值会变呢，因为虽然props不可变，但是this可变，this上的数据是可以被修改的，this.props的调用每次都会获取最新的props，而这正式React确保数据实时性的一个重要手段。

    函数组件的props会在执行的一瞬间就被捕获，而props不可变，因此可以充分确保从任何时候读取到的props都是最初捕获到的。函数组件正在地把数据和渲染绑定到了一起）

    类组件：封装和继承，大而全，***面向对象***的设计思想。

    函数组件：轻巧灵活，***函数式编程***的设计思想。

    函数组件更加契合React框架的设计理念，因为 ***UI=f(data)***，也更有利于逻辑拆分与重用的组件表达形式;

    React组件本身的定位就是函数，一个接收数据、吐出UI的函数。

    使用Hooks的好处

    • 告别难以理解的Class（this和生命周期）
    • 解决业务逻辑难以拆分的问题
    • 使状态逻辑复用变得更简单

    # Hooks不是万能的

    • 暂时还不具备完全为函数组件补齐类组件的能力：getSnapshotBeforeUpdate、componentDidCatch
    • 轻量和复杂的取舍：业务驱动
    • 使用层面有着严格的规则约束

    # Hooks的使用规则

    1. 只在React函数中调用Hook；
    2. 不要再循环、条件和嵌套函数中调用Hook；

    原则2：目的是要确保Hooks在每次渲染时都保持同样的执行顺序

    若不保证Hooks执行顺序，会报错：组件渲染的Hooks比期望中更少。因为***hooks 的渲染是通过“依次遍历”来定位每个 hooks 内容的。如果前后两次读到的链表在顺序上出现差异，那么渲染的结果自然是不可控的***。

    Hooks的正常运作，在底层依赖于顺序链表

    首次渲染：mountState的主要工作时初始化Hooks。

    关注源码的mountWorkInProgressHook方法

    function mountWorkInProgressHook() {
  // 单个Hook是以对象的形式存在的
  var hook = {
    memoizedState: null,
    baseState: null,
    baseQueue: null,
    queue: null,
    next: null
  };

  if (workInProgressHook === null) {
    // 这行代码每个 React 版本不太一样，但做的都是同一件事：将 hook 作为链表的头节点处理
    firstWorkInProgressHook = workInProgressHook = hook;
  } else {
    // 若链表不为空，则将 hook 追加到链表尾部
    workInProgressHook = workInProgressHook.next = hook;
  }
  // 返回当前的 hook
  return workInProgressHook;
}

    更新渲染：按顺序去遍历之前构建好的链表，取出对应的数据信息进行渲染。

    # 虚拟DOM

    本质：是JS和DOM之间的映射缓存

    形态：能够描述DOM结构及其属性信息的JS对象

    # 前端DOM操作的发展：

    静态页面+少量JS ----> JQuery ----> JQuery + '模板引擎（字符串拼接）' ----> 虚拟DOM；

    虚拟DOM和模板引擎在思想上存在递进关系；

    # 选择虚拟DOM真是为了更好的性能吗？

    在整个DOM操作发展里，主要矛盾不在于性能，而在于研发体验和工作效率。所以虚拟DOM是前端开发在追求更好的研发体验和效率而创造出的高阶产物。

    虚拟DOM并不一定能带来更好的性能，官方也没有把虚拟DOM作为性能层面的卖点对外输出过。

    虚拟DOM的优越之处在于，在提供了更高效的研发模式（函数式UI编程方式）的同时，仍然还保持一个还不错的性能。

    只有当页面修改少量DOM时，虚拟DOM才有性能上的绝对的优势。

    虚拟DOM除了差量更新外，批量更新也是在性能方面所作的重要努力。批量更新在通用虚拟DOM库里是由batch函数处理的。当有多次快速更新的情况下，前几次的更新是无用的。

    此时batch的作用是***缓冲每次生成的补丁集***。它会把收集到的多个补丁集暂存到队列中，再将最终的结果交给渲染函数，最终实现集中化的DOM批量更新。

    # 虚拟DOM的价值

    虚拟DOM的价值绝对不在性能上。

    虚拟DOM解决了两个关键问题：

    1. 研发体验和研发效率问题
    2. 跨平台的问题。虚拟DOM是对真实渲染内容的一层抽象。如果没有这层抽象，视图层和渲染平台会紧密耦合。有了这一层抽象，则可以实现一套代码多端运行。

    # 栈调和 和 Fiber调和

    调和 ：reconciler ；通过ReactDOM等类库，将虚拟DOM与”真实的“DOM同步的过程（将虚拟DOM映射到真实DOM的过程）。

    调和 !== diff ：调和是”使一致的过程“；diff 是 找不同的过程，是调和过程的一部分。

    React从大板块上将代码分为了 core、renderer和reconciler三个部分，其中reconciler(调和器)的工作包括组件的挂载、更新和卸载，其中更新阶段使用的就是对DIff的调用。

    # Diff策略的设计思想

    由于对两棵树找不同的算法复杂度O(n^3^)太复杂，所以react团队在降低算法复杂度的时候确立了两个前提：

    1. 若两个组件具有相同的类型，那么他们将拥有相同的DOM结构。
    2. 处于同一层级的一组节点，可以通过设置唯一的key，维持各个节点在不同的渲染过程中的稳定性。

    还总结出一个比较紧密的规律：DOM跨层级操作不多，同层操作是主流。

    # Diff逻辑

    把握3点：

    1. Diff算法性能突破的关键在于***”分层对比“***

    2. 类型一致的节点才有继续diff的必要性

    3. key属性的设置，可以帮我们尽可能重用同一层级内的节点

    # 1. 改变时间复杂度量级的决定性思路：分层对比

    react直接放弃跨层级节点比较，只针对相同层级的节点做对比。这样，一次遍历就能完成整棵树的对比，这是降低复杂度的一个重要设计。

    需要注意的是：虽然栈调和将传统的树对比算法优化为了分层对比，但***整个算法仍是递归形式***，分层递归也是递归。

    如果真的发生了跨层级操作，则将跨层级的组件全部销毁，在新的层级重建。销毁+重建代价昂贵，所以官方建议不要进行跨层级操作，尽量保持DOM结构的稳定性。

    # 2. 减少递归的一刀切策略：类型一致性决定递归的必要性

    参与diff的两个组件类型不一致，则放弃比较，直接替换旧节点。类型相同才会往更深层次diff。

    # 3. 重用节点：key属性帮助react记住节点

    key是用来帮助React识别哪些内容被更改、添加和删除。key需要写在用数组渲染出的元素内部，并且赋予一个稳定的值。稳定很重要，如果key发生了改变，React会触发UI的重渲染。是一个非常有用的特性。

    key试图解决的是同一层级下节点的重用问题。

    # setState到底是同步的还是异步的

    import React from 'react';

export default class App extends React.Component {
  state = {
    count: 0
  }
  increment = () => {
      console.log('increment setState前的count', this.state.count); // 0
      this.setState({
          count: this.state.count + 1
      });
      console.log('increment setState后的count', this.state.count); // 0
  }
  triple = () => {
      console.log('triple setState前的count', this.state.count); // 1
      this.setState({
          count: this.state.count + 1
      });
      this.setState({
          count: this.state.count + 1
      });
      this.setState({
          count: this.state.count + 1
      });
      console.log('triple setState后的count', this.state.count); // 1
  }
    reduce = () => {
        setTimeout(() => {
            console.log('reduce setState前的count', this.state.count); // 2
            this.setState({
                 count: this.state.count - 1
            });
            console.log('reduce setState后的count', this.state.count); // 1
        }, 0)
    }
    
   render() {
       return (
           <div>
           		<button onClick={this.increment}>点击加+1</button>
                <button onClick={this.triple}>点击+3？</button>
                <button onClick={this.reduce}>点击-1</button>
       		</div>
       )
   }
}

    setState是异步的，为了避免频繁re-render。

    异步的实现方式类似与Vue的$nextTick和浏览器的Event-loop：每来一个setState就放进队列内储存，直到最后一个setState，返回最新的state，更新视图。

    只有受react控制的setState才是异步的。

    # setState工作流

    enqueueSetState:

    1. 将新的state放入组件的状态队列里
    2. 调用enqueueUpdate，处理将要更新的对象

    **enqueueUpdate：**判断是否有批量创建/更新的组件

    1. 有，组件入dirtyComponent队列
    2. 没有，遍历dirtyComponent队列，更新组件

    batchStrategy是react内部管控批量更新的对象 ： 本质是一个锁管理器。控制更新是直接返回还是进入dirtyComponent等待下一次更新。

    # 同步现象的本质

    在异步的setTimeout内部，外部事务包裹的setTimeout函数执行setState时，isBatchingUpdates已经变为false，相当于锁管理器失去作用，这时setState就表现为同步。

  routes: Array<IRoute>;
}

  <Route key={path} path={path} component={Component}></Route>
);

const TransitionRoute: React.FC<TransitionRouteProps> = ({ routes }): any => {
  const location = useLocation();
  
  return (
    <TransitionGroup>
      {/*注意key，不能用location.key，或一直加载当前页面*/}

      </CSSTransition>
    </TransitionGroup>
  );
};

export default TransitionRoute;
